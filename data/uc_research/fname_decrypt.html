FName decrypt working, need help with FText for Item ESP names
Arc Raiders Item ESP showing "BP_PickupBase" for everything need help reading FText

Hey guys,

First of all, I don't know if what I'm posting is correct, it's what I found, so please don't feel offended if it doesn't meet your personal expectations.

Perhaps some offsets are also outdated.. I confirmed many using IDA and others using the SDK.

Been working on an external Item ESP for Arc Raiders for a while now. Positions are working fine, items show up on screen where they should be. The problem is every single item renders as "BP_PickupBase" no matter what it actually is rifle, medkit, ammo, whatever.

I was using FName, which is descriptive, and getting "BP_PickupBase" every time. I was only getting the class, not the item name. All items in the game are obviously instances of the same class, so they all share the same FName.

The actual item name (what the player sees in-game like "Assault Rifle MK-IV") seems to be stored as an FText inside a DataAsset. From what I can tell reversing FText::FromString and NSLOCTEXT in IDA, From what I could tell, the Ftext doesn't seem to be encrypted. It's just plain WCHAR behind a couple pointer dereferences. Very different from FName which has the whole SIMD+FNV multi-layer nightmare.

Looking for someone who's either done this already or can confirm my pointer chain theory before I keep banging my head against it.


What I have working
  • FName decryption (fully working the SIMD + FNV-32 + FNV-64 + XOR key table stuff)
  • World position via RootComponent+0x300 -> ComponentToWorld
  • Class identification using FName decrypt
    Code:
      auto name = GetActorFNameString(actor);

So basically everything works except I'm stuck showing the wrong name.


What I think the fix is

Instead of reading FName (class name), read the FText DisplayName from the item's DataAsset. The chain I came up with from RE:

Code:
Path A  via UIHoverData:
  Pickup + 0x540  ->  DataAsset*
UIHoverData + 0x20  ->  DataAsset*  (= Pickup + 0x560)
    DataAsset + 0xC8  -> FText 
      FText + 0x00  -> ITextData* 
        ITextData + 0x28  ->   WCHAR*   <- should be " Rifle XYZ" etc

Path B  via DefaultPickupDataAsset (fallback):
  Pickup + 0x488  ->  DataAsset*
    DataAsset + 0xC8  -> FText  -> ITextData + 0x28  -> WCHAR*

Path C  WorldItemBase:
  WorldItemBase + 0x898  --:>  DataAsset*
    DataAsset + 0xC8  -> FText  -> ITextData + 0x28  -> WCHAR*

FText layout from RE

Got this from decompiling FText::FromString (sub_12CAA0) and the NSLOCTEXT path (sub_101440). Both use the same vtable (off_AA4EC10) and same layout:

Code:
FText struct
  +0x00  ITextData*           
  +0x08  RefCountController*  
  +0x10  uint32 Flags          
  +0x14  padding

ITextData (what FText+0x00 points to):
  +0x00  vtable                off_AA4EC10
  +0x08  int32 state
  +0x10  WCHAR* Namespace     
  +0x18  WCHAR* Key            s
  +0x20  int32 Namespace.Num
  +0x24  int32 Key.Num
  +0x28  WCHAR* DisplayString  <-  plain text, NOT encrypted
  +0x30  int32 DisplayString.Num  
  +0x34  int32 DisplayString.Max  bufffer
FTextProperty::ElementSiz. Checked via the "FTextProperty" string xref -> sub_47F720.
